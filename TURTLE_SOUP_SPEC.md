# 海龟汤（Turtle Soup）

## 1. 项目概述

《海龟汤》是一个由 AI 主持的推理谜题游戏引擎。

玩家将面对一个表面荒诞或不完整的事件描述（即“汤面”），  
通过不断提出问题，逐步还原隐藏的完整故事（即“汤底”）。

AI 在系统中扮演：
- 谜题主持人
- 事实裁判
- 信息控制者
- 推理节奏管理者

本项目的目标不是制作固定谜题，而是构建一个：

> **可持续生成推理谜题并稳定主持问答过程的 AI 游戏系统**

---

## 2. 核心体验目标

### 2.1 玩家体验

玩家进入一局游戏后：

1. 获得一个简短但反常识的事件描述（汤面）
2. 通过提出 Yes / No / 无关 / 部分相关 类型问题进行推理
3. 不断逼近真相
4. 最终完整还原事件逻辑
5. 游戏结束并揭示完整故事

### 2.2 AI 主持体验

AI 必须：

- 严格掌握谜题真实答案（汤底）
- 不随意修改谜题事实
- 不因为玩家问法改变设定
- 控制信息释放速度
- 保持谜题可解性与逻辑一致性

---

## 3. 游戏本质模型

《海龟汤》不是聊天游戏，而是：

> **受控事实空间中的信息探索游戏**

系统必须区分三种信息层：

### 3.1 已公开信息（Public）

玩家一开始看到的“汤面”

例如：

> 一个男人走进餐厅，喝了一口海龟汤后自杀了。

### 3.2 隐藏真实信息（Truth）

完整故事：

- 人物关系
- 时间顺序
- 动机
- 关键事件
- 环境细节
- 情绪变化

### 3.3 可被揭示的信息（Discoverable）

玩家可以通过合理问题逐步获取：

- 部分事实
- 否定某些假设
- 锁定范围
- 排除干扰

---

## 4. 游戏流程设计

### 4.1 一局标准流程

1. 系统生成或加载谜题
2. 展示汤面
3. 玩家开始提问
4. 系统解析问题意图
5. 系统在 Truth 空间中判断：
   - 是否成立
   - 是否相关
6. 返回标准回答
7. 玩家持续推理
8. 玩家给出最终猜测
9. 系统判定是否正确
10. 公布完整汤底

---

## 5. 谜题结构设计

系统中的谜题应为结构化对象，而不是纯文本。

### 5.1 Puzzle 数据模型建议

```json
{
  "puzzleId": "string",
  "title": "string",
  "soupSurface": "string",
  "truthStory": "string",
  "facts": [],
  "misleadingPoints": [],
  "keyTriggers": [],
  "difficulty": "easy | medium | hard",
  "tags": []
}
```

### 5.2 facts（事实集合）

用于 AI 判断玩家问题。

```
{
  "factId": "string",
  "statement": "男人曾经遭遇船难",
  "importance": 0-10,
  "discoverable": true
}
```

------

### 5.3 misleadingPoints（误导点）

用于设计汤面的“反常识性”。

例如：

- 海龟汤是关键
- 自杀行为不是突发
- 场景真实但理解错误

------

### 5.4 keyTriggers（关键突破点）

玩家若问到：

- 是否与海难有关？
- 是否和过去经历有关？

游戏进度应明显推进。

------

## 6. AI 主持规则

### 6.1 回答类型限制

AI 必须只使用以下回答类型：

- 是
- 否
- 无关
- 部分相关
- 无法判断（极少使用）

禁止：

- 主动讲故事
- 补充背景
- 解释谜题逻辑
- 给提示（除非提示模式）

------

### 6.2 回答生成流程

1. 解析玩家问题
2. 转换为语义命题
3. 在 Puzzle.facts 中检索
4. 判断相关度
5. 输出标准回答

------

### 6.3 不允许行为

AI 不可以：

- 因为玩家提问方式不同改变谜题事实
- 动态生成新的真相
- 用文学发挥填坑
- 把问题当闲聊回答
- 提前剧透关键点

------

## 7. 游戏模式设计

### 7.1 单人模式（MVP 必做）

玩家 vs AI 主持人

### 7.2 多人模式（后续）

多人轮流提问
 共享推理进度

### 7.3 提示模式

可消耗提示点获得：

- 范围提示
- 排除提示
- 动机提示

------


------

## 9. 推理进度系统（推荐）

系统可通过以下方式衡量进度：

- 玩家触达关键 fact 数量
- 玩家问题质量
- 假设正确率
- 是否接近核心动机

用于：

- 提示触发
- 难度调节
- 游戏结束判断

------

## 10. 谜题生成系统（高级功能）

AI 可动态生成谜题，但必须满足：

### 10.1 可解性

- 必须存在合理推理路径
- 不依赖超自然逻辑（除特殊标签）

### 10.2 信息闭环

所有谜题必须：

- 动机成立
- 行为成立
- 时间线成立
- 环境成立

### 10.3 不允许

- “因为他疯了”式答案
- 纯巧合驱动结局
- 无法通过问句接近真相

------

## 11. MVP 功能要求

必须支持：

- 加载一个谜题
- 展示汤面
- 玩家连续提问
- AI 给标准回答
- 玩家提交最终猜测
- 系统判定正确或失败
- 展示完整汤底
- 存储问答记录

------

## 12. 技术模块建议

### Puzzle Engine

谜题加载与管理

### Question Parser

玩家问题语义解析

### Truth Validator

判断问题真假与相关性

### Host AI Layer

生成标准回答

### Session Manager

游戏局状态管理

### Persistence

存档与历史记录

------

## 13. 开发优先级

Phase 1：

- 固定谜题
- 单人问答
- Yes/No 判断
- 会话存储

Phase 2：

- 动态提示
- 推理进度条
- 多谜题切换

Phase 3：

- AI 谜题生成
- 多人模式
- 排行榜

------

## 14. 最终目标

构建一个系统，使玩家可以：

- 随时进入一局推理
- 与 AI 进行纯逻辑博弈
- 体验“真相逐渐浮出水面”的快感
- 每一局都是一次思维探险

这不是猜谜合集
 这是一个：

> **AI 推理主持引擎**
